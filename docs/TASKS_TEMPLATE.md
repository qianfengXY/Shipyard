# TASKS TEMPLATE

说明：

- Shipyard 按子任务执行，不直接执行模块本身。
- 一个模块下的全部子任务都完成，才表示该模块完成。
- 所有模块完成，才表示整份产品任务完成。
- `task_id` 必须全局唯一。
- `module_id` 必须全局唯一。

全局执行规则：

- 代码、产品文档、任务书、验收文档统一采用 Git 管理。
- 一个功能模块下的全部子任务完成后，执行一次模块级提交，方便后续按模块回滚。
- 如果产品文档、任务书或验收要求发生变化，更新文档后重新运行 Shipyard；默认从上次失败或未完成的子任务继续。
- 每个子任务都必须可独立验收，避免一个任务跨越多个功能模块。

示例格式：

## module-auth 账号与认证模块

depends_on: none

- [ ] auth-001 实现邮箱登录接口
- [ ] auth-002 实现登录页面与错误态
- [ ] auth-003 为登录流程补充自动化测试

## module-workspace 工作台模块

depends_on: module-auth

- [ ] workspace-001 实现工作台首页列表
- [ ] workspace-002 支持列表筛选与空状态
- [ ] workspace-003 补充工作台模块验收文档

编写建议：

- 每个子任务只描述一个明确交付结果。
- 子任务标题尽量写成“动词 + 对象 + 限定条件”。
- 如果某模块过大，先继续拆分为更小的子任务，再放进 TASKS。
- 如果模块依赖其他模块，使用 `depends_on:` 明确列出前置模块；多个依赖用逗号分隔。
- 如果需要让 Builder/Verifier 更稳定，给每个子任务补充对应的验收文本到 `docs/ACCEPTANCE_SPEC.md`。
